# SYSTEM STATE

Last updated: 2026-04-21 (CrUX field-data fallback for INP/CLS; executive summary caption reveal + widened container; metric-quality stage attenuation)

## Product Summary

Performance Intelligence is a SaaS platform that converts Lighthouse/PageSpeed performance data into business intelligence. It answers: where is revenue lost due to performance, which resources cause it, and what fixes recover the most revenue. Target users are founders and product owners; the product is revenue intelligence, not a developer tool.

## Completed Systems

- **Lighthouse/PageSpeed ingestion** — `GET /api/analyze`; PageSpeed Insights API (mobile + desktop). **CrUX field-data fallback for core vitals:** `app/api/analyze/route.ts` resolves LCP/CLS/INP/FCP from the Lighthouse lab audit first, then from the top-level **`loadingExperience.metrics[KEY].percentile`** (URL-level CrUX), then from **`originLoadingExperience.metrics[KEY].percentile`** (origin-level CrUX) via the module-scope **`readFieldPercentile(pageSpeedData, key)`** helper. CrUX stores CLS as integer × 100, so it is divided back to decimal before use. **Without this, INP is silently lost** — it is a field-only metric, so the lab audit `interaction-to-next-paint` usually has no `numericValue`. Snapshot writes store **`null`** (not `0`) when the resolved value is genuinely missing, so the dashboard renders `—` via `snapshot.inp != null ? ... : "—"` instead of a misleading `0 ms`. The top-level **`loadingExperience`** and **`originLoadingExperience`** are also preserved onto **`rawAudit`** (in addition to `lighthouseResult`) for downstream traceability. Dev log includes `sources: { lcp, cls, inp }` each tagged `"lab" | "field" | "missing"`.
- **Metric intelligence engine** — `lib/metricIntelligenceEngine.ts`; extracts LCP, FCP, TTI, TBT, CLS, Speed Index, TTFB, mainThread, bootupTime, unusedJs, unusedCss; outputs cleanMetrics, detailedMetrics, metrics_for_dashboard
- **Risk engine** — `lib/riskEngine.ts`; speed, UX, SEO, conversion, scaling risk scores (0–100); overall health score
- **Revenue impact engine (CORTEX v2)** — `lib/impactEngine/cortexV2.ts`; stage-based curves (VISUAL_ENTRY, INTERACTION, TRUST, INFRASTRUCTURE); outputs totalLeak, leakByMetric, diagnostics. **Diagnostic only:** `app/api/analyze/route.ts` always runs CORTEX after the headline path and attaches **`cortex_diagnostic`** on the analyze JSON (does not overwrite headline leak).
- **Behavior-based headline revenue at risk** — `lib/impactEngine/v2BehaviorModel.ts` (`computeBehaviorRevenueImpact`) via **`lib/headlineRevenueLeak.ts`** (`computeHeadlineRevenueLeak`, shared metric extraction). Same per-metric impact rules as before. **`GET /api/analyze`** and **`components/RevenueImpactWorkspace.tsx`** use this single helper so headline **`estimatedMonthlyLeak`** / **`leak_by_metric`** stay aligned (keys: `lcp`, `tbt`, `inp`, `cls`). Baseline resolution: profile with advanced inputs → `computeBaselineRevenue`; else profile `monthlyRevenue` when > 0; else default monthly revenue for the request. See `docs/DECISION_LOG.md` and `docs/REVENUE_MODEL_AUDIT.md`.
- **Revenue projection engine (driver attribution)** — `impactEngine/revenueProjectionEngine.ts`; `runImpactEngine`; produces `primaryDrivers` + raw stage scores using the V2 impact functions (LCP/CLS/TBT/INP; **`TTI`** uses lab `snapshot.tti` in seconds → ms via the TBT-shaped curve, falling back to TBT when TTI is missing). Its own opportunity range is **discarded** by the Workspace. **Revenue Impact** stage cards use **`lib/revenueStageDistribution.ts`** (metric severities on the performance snapshot).
- **Resource attribution engine** — `lib/impactEngine/revenueLeakCalculator.ts`; `attributeLeakToResources()`; USA square-root weighting; distributes metric leak across resources

- **Financial forensic audit** — Metric drawer Financial tab (Pro); per-resource revenue, confidence, Why/Recommended fix/Expected improvement; `/financial-report` page (Pro or demo)

## Current Features

- Run performance analysis (URL on home → `/api/analyze` → dashboard or report). **Main JSON does not include competitor analysis** (competitor feature is separate).
- **Revenue Stability Monitoring** on the project dashboard: current status, directional “what changed,” previous vs current revenue at risk (exact range strings), optional alert, Pro-only minimal direction history (no metric numbers in history lines).
- View overall health score and the **Impact Breakdown** card — "Where performance is impacting your business" — across Load Experience / Interaction Experience / Search Visibility / Conversion Readiness / Scalability Risk, each with a business-language explanation, an impact label (High revenue risk / Moderate impact / Low impact), and a full-width **"View root causes →"** button that scrolls to the Metrics Responsible section. No numeric scores are shown.
- View estimated monthly revenue leak and leak by metric (behavioral headline; per-metric drawers use **`resolveLeakForMetricDrawer`** so TTI pools TBT+INP when there is no `tti` dollar slice)
- Open metric drawers: Technical tab (explanation, impact) and Financial tab (Pro: resources, revenue attribution, recommendations)
- See contributing resources (Pro: URLs, revenue, confidence; Free: grouped categories only)
- See recommended fixes (Pro, Financial tab only): Why / Recommended fix / Expected improvement
- View Revenue Recovery Opportunities section (Pro: full cards with action titles, Improves badges, revenue or "Revenue recovery potential"; Free: LockedRevenueRecovery with blurred preview + CTA)
- View Priority Fixes (Pro: revenue-impact cards with "Recover up to $X/month across Y fixes" header + Additional Performance Optimizations; Free: blurred preview + CTA). Only items with measurable revenue appear in Priority Fixes; $0 items appear in Additional Performance Optimizations. Click "Open Fix Guide" to open the metric drawer on the Financial Forensic Audit tab (`?metric=...&tab=forensic`).
- **Competitor comparison** — Modal → `GET /api/competitor-analysis?url=...&competitorUrls=...` (optional `baselineRevenue`). UI: headline → **Biggest issue** banner (when Behind) → **Snapshot table** (Factor, Position, Impact, Why it matters; sorted; highlighted row) → edge/shortfall → What this means → Decision / Why / Confidence → Business impact + revenue at risk → Opportunity → Actions + See Priority Fixes → Compare again.
- Access `/financial-report` with `?demo=1` or Pro (else LockedRevenueRecovery)
- Refine revenue analysis (RefineRevenueModal); Revenue Impact card; Growth Impact Roadmap
- Run **Revenue Impact** workspace (`/dashboard/[projectId]/revenue-impact`) — page header is "Revenue Impact" (no "Simulation"). Section is now structured as a **financial decision system**:
  - **Hero**: "Revenue at risk: $LOW – $HIGH /month" + "≈ X% – Y% of your revenue at risk" (hidden when baseline is missing/zero) + recovery framing line ("A large portion of this loss is recoverable.") + behavioral primary insight derived from the highest-loss stage (e.g. "Most revenue is lost before users see your page.") + actionable confidence (`Confidence: High/Medium/Low` + one-line meaning per level — High → "Safe to prioritize this for immediate impact.") + neutral trust line ("Based on performance patterns and user behavior impact.").
  - **Stage breakdown** ("Where users are dropping"): three cards using behavioral headings (Before users see your page / When users try to interact / When users decide to convert). **No sub-labels** — the previous generic, technical sub-labels (Load speed issue / Interaction delay / Layout instability) have been removed. One executive-style sentence per card is selected deterministically from a controlled `STAGE_MESSAGE_POOL` (5 sentences per pool, indexed by `Math.floor(expectedLoss || 1) % pool.length` where `expectedLoss = (low + high) / 2`) — non-repetitive across projects, fully stable across renders, no metric names, no hedging language, no `Math.random()`. Each card shows ONLY: heading, the single sentence, the monthly value labelled **"Recoverable impact"**, and the "Start here" badge on the highest stage. Internal stage names (`Landing` / `Activation` / `Conversion`) are never surfaced.
  - **Metric-driven stage distribution (UI-only)**: `lib/revenueStageDistribution.ts` maps snapshot LCP / INP+TBT (INP preferred, TBT fallback, optional lab TTI) / CLS to three funnel weights with Web Vitals-style severity bands, interaction-dominance rules when TBT or INP is bad, and a cap so **good CLS (≤0.1)** cannot exceed **10%** of total weight. Per-card **Recoverable impact** = share of `expectedRecoverable = ((opportunityLow + opportunityHigh) / 2) × recovery.avg` with sum within **≤2%** of target. Good CLS forces Conversion to non-blame copy (no layout-shift blame) and excludes it from "Start here" when gated; any stage under `STAGE_LOW_IMPACT_PERCENT` (12%) share still uses non-blame lines. `highestStageIndex` skips low-impact–gated stages when a better alternative exists.
  - **Mode-aware presentation-layer derivation (UI-only, single source of truth)**: `RECOVERY_FACTORS` (Conservative `{min: 0.5, max: 0.6, avg: 0.55}`, Balanced `{min: 0.6, max: 0.75, avg: 0.675}`, Aggressive `{min: 0.75, max: 0.9, avg: 0.825}`) drives recoverable bounds, projected revenue, and the `expectedRecoverable` target for stage dollars. Each render derives from the engine's mode-independent `totalLoss` (= `result.opportunityRange.expected`) plus the current `sensitivityMode`: at-risk range = `totalLoss × RANGE_BOUNDS_BY_MODE[mode]` (capped at baseline); recoverable bounds = `opportunityLow × recovery.min` / `opportunityHigh × recovery.max`; projected = `currentRevenue + recoverable{Low,High}`; per-stage recoverable amounts are `distributeRecoverableAcrossStages` (sum ≈ that `expectedRecoverable` within ≤2%). Switching estimation style updates the at-risk range, recoverable range, projected revenue, stage values, and CTA in the same render — no engine re-run. Backend, persistence, and `result.opportunityRange` are untouched.
  - **CTA**: "Recover up to $X/month →" where X = `reconcileMaxRecoverableCta` (base `round(opportunityHigh × recovery.max)`, then **≥ each stage** and **≥ sum(stages)**, capped at `validated.totalLoss`). Falls back to "See what to fix to recover this revenue →" when zero. Links to `/dashboard/{projectId}#action-priority-list`.
  - **Collapsibles**: "Projected revenue after fixes" + "How this estimate works" (rewritten plain-language: slow loading → drop-offs, interaction delays → engagement, layout shifts → trust; under 5 lines; no LCP/CLS/INP names) — collapsed by default.
  - **Confidence override** based on input completeness vs the chosen business model's required keys (Low / Medium / High).
  - **Performance snapshot** grid remains as supporting evidence below the collapsibles.
  - Realistic-input safeguards (soft warnings + internal caps for extreme conversion values) and auto-corrected mathematical invariants (`validateRevenueModel`) are unchanged.
  - Banners use "Re-run the estimate." and the run button uses "Estimating…" / "Calculate revenue impact" — words like "simulation", "stage-based model", and "calculation engine" are never surfaced in user copy.
- Demo unlock: UpgradeModal sets `localStorage.demoPro` and opens financial-report; dashboard syncs demoPro from localStorage

## Known Issues

- **Legacy snapshots still have `inp = 0`** — Snapshots written before the CrUX field-data fallback landed store `inp = 0` for URLs where lab INP was missing (because the old pipeline used `|| 0`). New snapshots correctly store the field-data value (URL-level → origin-level → `null` when no CrUX coverage). Re-analyze the project to replace an old snapshot; there is no automatic backfill of historical rows.
- **userPlan not set by API** — `app/api/analyze/route.ts` returns `userAuthenticated` only; does not set `userPlan`. Pro access is effectively demo-only unless `userPlan` is added to stored payload elsewhere.
- **lib/recommendationEngine.ts unused** — `generateSimpleRecommendation()` is not used; drawer uses only `getMetricResourceRecommendation()` in DetailedMetricsBoard.
- **Duplicate recommendation logic** — RevenueRecoveryOpportunities has its own `getRecommendationCopy()`; DetailedMetricsBoard has `getMetricResourceRecommendation()`. Two sources of recommendation copy.
- **Revenue sometimes $0** — When attribution yields $0, UI shows "Revenue recovery potential" (fixed in Revenue Recovery); metric-level leak can still be zero for some runs.
- **In-drawer upgrade CTA only on Financial tab** — Free users see "Unlock Performance Intelligence" only when Financial tab is selected; Technical tab has no CTA.
- **getGenericCategoryForResource** — Imported from `lib/metricDrawerMonetization` in DetailedMetricsBoard but never used.
- **Legacy competitor HTTP routes** — `app/api/compare/route.ts` and `app/api/competitive-intelligence/route.ts` still exist; primary competitor UX uses `/api/competitor-analysis` + dashboard.
